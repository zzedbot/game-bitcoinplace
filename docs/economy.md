### 1.2 赛季制设计 ⭐ 核心机制

**一个赛季 = 60 天**，分为三个阶段：

```
┌──────────────────────────────────────────────────────────────┐
│  第 1-49 天        第 50-56 天         第 57-60 天            │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │  挖矿期     │  │  自由期     │  │  停滞期     │          │
│  │  Mining     │  │  Free       │  │  Frozen     │          │
│  │             │  │             │  │             │          │
│  │  • 7 天减半   │  │  • 无新产出  │  │  • 禁止修改  │          │
│  │  • 产出 2100 万│  │  • 自由交易  │  │  • 定格画面  │          │
│  │  • 随机分配  │  │  • 集中创作  │  │  • 赛季结算  │          │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
│                                                              │
│  ◄────── 49 天 ──────►◄──── 7 天 ────►◄──── 4 天 ────►       │
│                                                              │
│                  一个赛季 = 60 天                              │
└──────────────────────────────────────────────────────────────┘
```

| 阶段 | 时间 | 时长 | 核心特征 |
|---|---|---|---|
| **挖矿期** | 第 1-49 天 | 49 天 | 7 天减半周期，产出 2100 万色块 |
| **自由期** | 第 50-56 天 | 7 天 | 无新产出，自由交易/创作/冲刺 |
| **停滞期** | 第 57-60 天 | 4 天 | 画布冻结，禁止修改，赛季结算 |

**赛季结束后：**
- 画布定格，生成赛季纪念 NFT/截图
- 排行榜结算，发放奖励
- 新赛季重新开始（画布重置或扩展新区域）

### 1.3 减半周期详细计算（方案 A：快速版）

**目标：49 天产出 2100 万色块**

**计算过程：**
```
总窗口数 = 49 天 × 24 小时 × 6 个/小时 = 7,056 个窗口
周期数 = 49 天 ÷ 7 天 = 7 个减半周期
每周期窗口数 = 7 天 × 24 × 6 = 1,008 个窗口

设初始产出为 X，根据等比数列求和：
X × 1008 + X/2 × 1008 + X/4 × 1008 + ... + X/64 × 1008 = 2100 万
X × 1008 × (1 + 1/2 + 1/4 + 1/8 + 1/16 + 1/32 + 1/64) = 2100 万
X × 1008 × 1.984375 = 2100 万
X ≈ 10,521

取整：初始产出 = 10,500 色块/窗口
```

**完整产出表：**

| 周期 | 时间范围 | 每窗口产出 | 日产出 | 周期产出 | 累计产出 | 完成度 |
|---|---|---|---|---|---|---|
| 0 | 第 1-7 天 | 10,500 | 151,200 | 1,058,400 | 1,058,400 | 5.04% |
| 1 | 第 8-14 天 | 5,250 | 75,600 | 529,200 | 1,587,600 | 7.56% |
| 2 | 第 15-21 天 | 2,625 | 37,800 | 264,600 | 1,852,200 | 8.82% |
| 3 | 第 22-28 天 | 1,312 | 18,893 | 132,250 | 1,984,450 | 9.45% |
| 4 | 第 29-35 天 | 656 | 9,446 | 66,125 | 2,050,575 | 9.76% |
| 5 | 第 36-42 天 | 328 | 4,723 | 33,062 | 2,083,637 | 9.92% |
| 6 | 第 43-49 天 | 164 | 2,362 | 16,531 | **2,100,168** | 10.00% ⚠️ |

⚠️ **问题发现：** 按照标准减半公式，7 个周期后只能产出约 210 万，远低于 2100 万！

**原因：** 等比数列求和收敛于 2 倍首项，即 `X × 1008 × 2 ≈ 210 万`

X × 1008 × (2 - 1/2^(n-1)) = 2100 万

若 n=20 个周期（140 天，超出 49 天限制）：
X × 1008 × 1.999998 ≈ 2100 万
X ≈ 10,417

### 2.2 各阶段详细规则

#### 阶段 1：挖矿期（第 1-49 天）

**核心规则：**
- 每 10 分钟为一个"区块窗口"
- 每日 144 个窗口（24 小时 × 6 个/小时）
- 7 天为一个减半周期，共 7 个周期
- 色块通过在线随机分配给活跃用户

**产出曲线：**

| 周期 | 时间 | 每窗口产出 | 日产出 | 周期产出 | 累计 | 完成度 |
|---|---|---|---|---|---|---|
| 0 | 第 1-7 天 | 10,417 | 150,005 | 10,500,336 | 10,500,336 | 50.00% |
| 1 | 第 8-14 天 | 5,208 | 74,995 | 5,249,664 | 15,750,000 | 75.00% |
| 2 | 第 15-21 天 | 2,604 | 37,498 | 2,624,832 | 18,374,832 | 87.50% |
| 3 | 第 22-28 天 | 1,302 | 18,749 | 1,312,416 | 19,687,248 | 93.75% |
| 4 | 第 29-35 天 | 651 | 9,374 | 656,208 | 20,343,456 | 96.87% |
| 5 | 第 36-42 天 | 325 | 4,680 | 327,600 | 20,671,056 | 98.43% |
| 6 | 第 43-49 天 | 325 | 4,680 | 328,944 | 21,000,000 | 100.00% |

**用户行为预期：**
- **第 1 周**：大量用户涌入，争夺 50% 的资源
- **第 2-3 周**：产出减半，投机者开始囤积
- **第 4-5 周**：稀缺性显现，拍卖行价格飙升
- **第 6-7 周**：FOMO 情绪，最后冲刺

#### 阶段 2：自由期（第 50-56 天）

**核心规则：**
- ❌ 无新色块产出
- ✅ 拍卖行正常交易
- ✅ 已拥有色块可自由修改颜色
- ✅ 允许用户间转账/赠送

**运营活动：**
- **艺术创作周**：鼓励用户完成大型作品
- **交易冲刺**：手续费优惠（如 2.5% 而非 5%）
- **最后机会**：未使用染色权的最后交易窗口

**用户行为预期：**
- 集中创作：艺术家用囤积的色块完成作品
- 价格发现：市场形成最终价格共识
- 清理库存：散户出售剩余染色权

#### 阶段 3：停滞期（第 57-60 天）

**核心规则：**
- ❌ 禁止任何染色操作
- ❌ 禁止修改颜色
- ✅ 拍卖行可继续交易（所有权变更，但不可使用）
- ✅ 只读模式浏览画布

**赛季结算：**
- 生成赛季纪念截图/NFT
- 排行榜结算（持仓量、交易额、活跃度）
- 发放赛季奖励
- 用户投票选出"最佳作品"

**用户行为预期：**
- 回顾反思：欣赏整个赛季的创作
- 社交传播：分享自己的作品和成就
- 期待新赛季：讨论下赛季策略

## 3. 分配机制设计

### 3.1 在线用户随机分配

**基础算法：**
```python
def distribute_rewards(online_users, reward_count):
    """
    online_users: [{user_id, online_minutes, activity_score}, ...]
    reward_count: 本窗口应产出的色块数
    """
    # 计算权重
    weights = []
    for user in online_users:
        # 在线时长权重（最多 2 倍，60 分钟封顶）
        time_weight = min(user.online_minutes / 60, 2.0)
        
        # 活跃度权重（基于历史行为）
        activity_weight = 1.0 + user.activity_score / 100.0
        
        # 持仓权重（已有染色权数量，鼓励分散）
        holdings_weight = 1.0 / (1.0 + user.color_rights_count / 10)
        
        weight = time_weight * activity_weight * holdings_weight
        weights.append(weight)
    
    # 加权随机抽样
    winners = weighted_sample(online_users, weights, reward_count)
    return winners
```

### 3.2 权重参数建议

| 因素 | 权重公式 | 说明 |
|---|---|---|
| 在线时长 | `min(分钟数/60, 2.0)` | 60 分钟以上不再增加 |
| 活跃度 | `1.0 + 活跃度/100` | 基于历史染色、拍卖行为 |
| 持仓量 | `1.0 / (1 + 持仓数/10)` | 鼓励分散，防止垄断 |
| 新用户奖励 | `1.5` (前 7 天) | 注册 7 天内额外 50% 概率 |

### 3.3 防作弊机制

**问题：** 用户可能开多个账号刷奖

**解决方案：**

1. **设备指纹**
   - 收集设备信息（浏览器指纹、IP、时区等）
   - 同一设备最多 3 个账号享受权重

2. **IP 限制**
   - 同一 IP 最多 5 个账号
   - 数据中心 IP 标记为高风险

3. **行为检测**
   - 挂机检测：无鼠标/键盘活动超过 30 分钟，权重减半
   - 脚本检测：规律性操作模式标记

4. **验证码挑战**
   - 中奖后需完成简单验证码才能领取
   - 高频中奖用户触发更严格验证

## 4. 拍卖行经济

### 4.1 定价机制

**货币系统：** 单一货币 - 像素币 (Pixel / PX)

| 属性 | 说明 |
|---|---|
| 名称 | 像素币 (Pixel) |
| 符号 | PX |
| 汇率 | 1 PX = 1 人民币 (CNY) |
| 最小单位 | 1 PX (支持小数，精确到 0.01) |
| 获取方式 | 在线奖励、活动、充值、交易所得 |
| 用途 | 购买染色权、拍卖交易、手续费、道具购买 |

**优势：**
- ✅ 简单直观，用户无需换算
- ✅ 降低理解门槛，新手友好
- ✅ 便于定价和价格发现

### 4.2 手续费设计

| 操作 | 费率 | 说明 |
|---|---|---|
| 挂单 | 0% | 免费挂单 |
| 成交 | 5% | 向卖方收取（像素币） |
| 提现 | 2% | 像素币→法币/加密货币 |
| 转账 | 1% | 用户间转账 |

**手续费用途：**
- 3% 进入平台金库（运营）
- 2% 进入奖池（特殊活动奖励）

### 4.3 价格发现机制

**问题：** 早期如何形成合理价格？

**方案：**

1. **官方指导价**（前 30 天）
   - 根据位置（中心 vs 边缘）给出建议价
   - 根据相邻坐标成交价动态调整

2. **拍卖优先**（而非固定价）
   - 前 1000 笔交易强制拍卖
   - 形成价格发现

3. **价格地板**
   - 设置最低交易价（如 100 PX）
   - 防止恶意倾销

### 4.4 稀有坐标溢价

**影响价格的因素：**

| 因素 | 溢价倍数 | 说明 |
|---|---|---|
| 矿区编号 | 1.0-2.0x | 创世矿区 (#1) 最贵 |
| 位置 | 1.0-5.0x | 中心区域更贵 |
| 相邻关系 | 1.0-10.0x | 连片土地溢价 |
| 历史意义 | 1.0-100.0x | 第一个染色的坐标等 |
| 特殊颜色 | 10.0-100.0x | 金色/荧光色 |

## 6. 风险与对策

### 6.1 经济风险

| 风险 | 描述 | 对策 |
|---|---|---|
| 通胀 | 早期产出过高，货币贬值 | 设置价格地板、增加消耗场景 |
| 通缩 | 后期产出过低，流动性枯竭 | 引入二手交易、租赁机制 |
| 垄断 | 大户囤积居奇 | 持仓权重递减、反囤积税 |
| 投机 | 价格剧烈波动 | 涨跌停限制（±20%/天） |

### 6.2 技术风险

| 风险 | 描述 | 对策 |
|---|---|---|
| 机器人 | 脚本刷奖 | 设备指纹、行为分析、验证码 |
| DDoS | 攻击导致服务中断 | CDN、限流、多活部署 |
| 数据库瓶颈 | 2100 万坐标查询慢 | Redis 缓存、分库分表 |
| WebSocket 压力 | 10 万并发连接 | 水平扩展、消息队列 |

### 6.3 运营风险

| 风险 | 描述 | 对策 |
|---|---|---|
| 活跃度下降 | 减半后用户流失 | 定期活动、社区运营 |
| 内容违规 | 用户绘制不当内容 | 审核机制、举报系统 |
| 法律风险 | 涉及虚拟币交易 | 明确用户协议、禁止法币交易 |
